Основные изменения в коде:
В рамках работы программа из лабораторной №1 была модифицирована для параллельного выполнения на графическом процессоре с использованием технологии CUDA.
Генерация входных матриц вынесена в отдельный Python-скрипт, а основное вычисление выполняется в CUDA-программе main.cu.
Для проверки корректности результата используется отдельный скрипт verify.py, который сравнивает результат GPU-вычислений с эталонным CPU-умножением.

Для распараллеливания использован стандартный подход с двумерной сеткой блоков. Каждый поток вычисляет один элемент результирующей матрицы.
Размеры блока задаются через параметры командной строки, что позволяет экспериментально сравнивать разные конфигурации сетки.
Для измерения времени выполнения используется CUDA events, что позволяет получить именно время GPU-вычислений без накладных расходов на запуск внешнего процесса.

Отдельно реализован скрипт run_bench.py, который автоматически генерирует матрицы, запускает вычисления для разных размеров и конфигураций блоков,
проверяет результат и сохраняет данные в CSV-файл для последующего анализа.

Примечание:
Инфраструктура: использовалась Windows-среда с установленным NVIDIA CUDA Toolkit 13.1 и компилятором nvcc.
Эксперименты проводились для размеров матриц 200, 400, 800, 1200, 1600 и 2000. Проверка корректности для всех запусков завершилась со статусом SUCCESS.

В качестве конфигураций сетки использовались блоки 8x8, 16x16 и 32x32. Такой набор позволяет сравнить производительность
при разных уровнях распараллеливания и загрузки потоков.

Анализ ускорения и эффективности:
Для анализа удобно взять размер N=2000 и сравнить конфигурации блоков относительно варианта 8x8.

Формулы:

S=T8x8/Tbx×by

E=S/1×100%

Так как здесь сравниваются не разные числа процессов, а разные конфигурации сетки,
эффективность можно интерпретировать как относительное улучшение по сравнению с базовой конфигурацией 8x8.

Конфигурация	Время (мс)	Ускорение относительно 8x8
8x8	            15.897	    1.00x
16x16	        14.752	    1.08x
32x32	        15.460	    1.03x
Для N = 2000 лучшую производительность показала конфигурация 16x16. Это объясняется тем,
что размер блока оказался удачным компромиссом между числом активных потоков, загрузкой SM и накладными расходами на планирование.
